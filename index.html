My Internship @ Samsara

An Unconventional Start

With the unforunate consequences of the COVID-19 virus, my internship with Samsara was remote. I was still extremely thankful for the opportunity since I had previously heard of how amazing the engineers and internship experience was at Samsara.

Goals for the internship

  • To continue learning more software engineering technology to improve my technical skills. With some other internships under my belt, I had a better picture of what the day-to-day of a software engineer was like. A main goal for my internship with Samsara was to be able to learn new technologies and taking ownership of a project.
  • Making an impact on Samsara and its users. I am especially motivated when I know that my project is making an impact on the company or end users. I hoped to be able to take ownership of a project, especially as a full-stack engineering intern. I wanted to learn from others while also being able to work independently to create something meaningful.
  • Learning and collaborating. I wanted to get to know other engineers, product managers, designers, and more! I wanted to collaborate with individuals from my team or outside my team, and I am motivated to work with different people to leran from them. At a company at Samsara's size, I wanted to get to know others and contribute to a healthy and thriving engineering culture.

My Project

Problem

At Samsara, their goal is to make it easier for fleet companies easily manage their trucks and drivers. Whenever unsafe events, such as harsh braking or collisions, come into Samsara's system, Samsara allows their customers to assign coaches to those safety events. However, especially with large enterprise customers, it may be a tedious process to have to manually assign those coaches. It can take time, effort, and be the bottleneck to an efficient coaching workflow.

Purpose

The purpose of my project was to platform where drivers can be given specific and explicit coach assignments. From there, we can use those assignments to then automatically assign coaches to safety events that come in.

Process

Since this was a full-stack project, there were some UI components and some backend logic that had to be implemented.
I was also very lucky to have support from my mentor, manager, and other team members. I was also able to work closely with the product manager and designer for my team.

  • Design Document
    • I kicked the project off by reviewing my design document. This was a document illustrating the purpose, stakeholders, the architecture, design, dependencies, security concerns, etc. involved in my project.
  • User Interface
    • Implementing workflows and designs created by designers.
    • Participating in design review sessions to gather/give feedback.
  • Backend
    • Spinning up my own database and microservice using Samsara's infrastructure.
    • Using Go to learn about RPC endpoints.
  • Testing
    • I constantly had to test my code by writing unit tests, testing changes locally and on sandbox, or using internal tools to query databases for any changes.
  • Customer Calls

My Internship @ Square

Getting settled in

When I heard that I got an offer from Square, I was super excited! Itâ€™s also so awesome to get an offer to work at such a top-notch but still small tech company. But I had to go to Atlanta. I barely knew anything about this place. But I knew that opportunities like these were hard to come by, so I decided to give it a shot.

When I first got here, things were so hectic. I had to get my room cleaned, go to Walmart to buy everything, make sure I fed myself, and prepared myself for work the next day. I was one of the five interns in the Atlanta office, and the only intern on the Payment Acceptance team. My team basically handled the many payments that come into Square every second. Sounds important huh? Especially knowing that a big part of Square is the POS that takes all the credit card payments. On my first day, I met with my manager about my goals for the internship and looked over some UI code.

Goals for the internship

  • Getting a feel for what it is like to work at a tech company. This was my first real internship, so learning about the best engineering practices, what it looks like to write good documentation and code, how to handle code review, all that good stuff.
  • Learning about the engineering that happens at Square specifically. At Square One (Square's orientation), I learned so much about the awesome technology that help accomplish Square's mission of empowering the underprivileged. Whether it is POS (Point of Sale), Square Cash, Captial, or anything else, it is so cool that one company can have so many subdivisions/businesses to serve its mission.
  • Doing full-stack development! My previous experience included a lot of middleware/frontend UI work, and I really wanted to be exposed to the magic that happens in the backend. I also still love web dev because that is something that I've been exposed to since my senior year of high school, so that would be interesting to keep exploring.

I was very lucky to have a project scoped out for me that was pretty much perfect in terms of what I'd work on.

My Project

Problem

At Square, whenever there are changes, payment processing partners want to ensure that the changes correct. In order to do this, they send test cases for "Certification." However, this certification process can actually be very difficult. It requires over ten steps and can be very time-consuming and complicated to understand

Purpose

The purpose of my project was to create a tool an internal tool that the Payment Acceptance team owns. This tool would allow users to create new batches (batches are groups of payments that come into Square) that can be sent out so that we can get back batch fies about the payments in the batch. Those batches are normally stored and encrypted somewhere and then sent to the payment processing partner, so creating a tool that can get back the batch files unencrypted and not send it to the payment processing partners (since these are only for testing purposes and we do not actually want the partners to process these test payments) would make the certification process much easier!

Process

Since this was a full-stack project, there were some UI components and some backend logic that had to be implemented.
I was also very lucky to have support from my mentor and manager. My mentor was my first point of contact / check-in if I had any questions, and I also met with my manager biweekly to discuss any problems/blockages

  • Design Document
    • I kicked the project off by writing a design document. This was a document illustrating the purpose, stakeholders, the architecture, design, dependencies, security concerns, etc. involved in my project.
  • User Interface (so glad that I had a few warmup assignments to improve the existing UI of the internal interface because that helped me understand what was happening and how I can make changes / add things to the interface)
    • Adding a new tab to the internal tool so that someone can use this tool
    • Creating filters and tables so that users can view the recent payments so that they know what payments they want their "test batch" to have
    • Displaying the payments that are added to the new batch
  • Backend
    • Lerning about RPC endpoints that are in the internal tool and learning how to create new classes for new RPC endpoints
    • Learning about dependency injections (Guice) and Data Defintion Languages (DDL - Protocol Buffers)
      • I also struggled with this a lot at first but gained a better understanding of these as I worked on a warmup task and as I continued to work through this project.
    • Learning about the payment flow technology that happen at Square, and make modifications to them so that I can create my tool.
      • Super confusing when I had no clue what was going on, but eventually got the hang of things :)
  • Testing
    • I constantly had to push my code to staging to test out changes.
    • Also wrote a few test files for the backend code that I added / modified.

Final Product & Takeaways

During the last few weeks, pull requests were created, code was reviewed, changes were made, and finally, my code was merged into ~master~! This tool is expected to be used in staging (and production if necessary).

During my last week, I had a final 30-minute presentation to the Payment team in Atlanta, along with a few other folks from the San Francisco office. I also presented a 5-minute demo during the bigger demos meeting between all Payment teams at Square.

I was overall really glad that I was able to complete the project in time and create a tool that would save Square teams time and frustration. Besides learning about the Square payment flow and all the technical work that goes into maintaining the services, I also felt like I was able to improve my presentation soft skills! I am so grateful that my team was so supportive throughout the entire internship, always being there to answer my questions and make my internship experience comfortable but also full of learning.
